# Design Variants — 서로 다른 가설의 시안 N종

## 포지션

UX 파이프라인의 "발산" 단계다: `product-brief`(PRD) → `design-reference`(레퍼런스 보드) → **design-variants(이 스킬)** → `working-prototype`(확정안 구현).
프로토타입은 비싸고 시안은 싸다 — 갈라볼 수 있는 마지막 싼 지점에서 가설을 가르는 것이 이 스킬의 존재 이유다.

## 제1원칙: 화장(cosmetic) 변주 금지

시안 3개가 색·여백·컴포넌트 스타일만 다르면 실패다. 시안은 **믿음(가설)이 달라야 한다** — 사용자 행동, 정보 구조, 첫 화면, 첫 행동, 무엇을 우선하는가가 달라야 한다.
검증법: 시안 라벨을 가리고 화면을 섞었을 때, 어느 화면이 어느 시안인지 구분할 수 있어야 한다. 구분이 안 되면 같은 시안의 스타일 변주일 뿐이다.

두 번째 원칙: **확정된 결정은 다시 열지 않는다.** 이미 결정된 것(예: 확정된 진행 표시 방식, 확정된 정책·카피 톤)은 전 시안 공통으로 깔고, **불확실성이 가장 큰 지점에서 가른다.** 레퍼런스 보드에 "선례 없음"으로 표시된 영역이 있다면 그곳이 1순위 분기 지점이다 — 선례가 있는 화면은 조립 문제라 갈라봐야 얻는 게 적고, 선례가 없는 화면은 어느 방향이 맞는지 모르기 때문에 갈라보는 가치가 크다.

## 입력 수집

세 문서를 찾는다. 일부만 있어도 진행하되, 빠진 것은 아래대로 처리한다.

| 입력 | 있으면 | 없으면 |
|---|---|---|
| PRD / 문제 브리프 | 요구사항·사용자 스토리·우선순위를 분기 재료로 사용 | 핵심 문제·타겟 사용자·꼭 들어가야 할 화면을 사용자에게 묻는다 (2~3개 질문). 발굴부터 필요해 보이면 product-brief를 제안 |
| 레퍼런스 보드 (특히 핸드오프/방향 분기 섹션) | 거기 정리된 방향 분기를 그대로 출발점으로 — 이미 계산된 발산을 재발명하지 않는다 | design-reference 실행을 먼저 제안. 사용자가 생략을 원하면 진행하되, 보드에 "레퍼런스 미검증" 명시 |
| 디자인시스템 | 토큰(색·radius·타이포)·Voice·금지 규칙을 프레임에 그대로 적용 | 그레이스케일 + 시스템 폰트 와이어로 그린다. 임의 브랜딩을 발명하지 않는다 |

익명화·법적 제약(예: 실명 금지, 실서비스 카피 복제 금지)이 입력 문서에 있으면 보드 전체에 적용한다.

## Phase 1 — 방향 도출 (텍스트, 컨펌 지점)

### 1. 설계 문제 정의
한 단락으로: 사용자는 누구이고(상황 변수 포함), 무엇을 하려 하고, 무엇이 성공이며, **가장 큰 불확실성이 무엇인가**. 마지막 항목이 분기의 씨앗이다.

### 2. UX 축 선택 (2~4개)
시안을 가를 축을 고른다. 축은 "어느 쪽이 맞는지 현재로선 알 수 없는 양극"이어야 한다. 자주 쓰는 축:

- **문제 진단**: 같은 불만의 원인을 다르게 본다 (예: 준비 부재 vs 안내 부재 vs 예측 불가) — 가장 강한 축. 가능하면 이걸 1축으로
- **타겟 상황 우선**: 어떤 상황의 사용자를 1순위로 최적화하는가
- **개입 강도**: 현행 보존 ↔ 정면 재설계 (이해관계자 설득·구현 비용 협상에 유리)
- 탐색형 ↔ 가이드형 / 콘텐츠 우선 ↔ 행동 우선 / 신뢰 구축 ↔ 속도 / 수동 제어 ↔ 자동화 / 검색 우선 ↔ 추천 우선

마땅한 축이 안 보이면 폴백: **A 익숙한 기본형 / B 전환·과업 최적형 / C 차별화 과감형**. 단 이는 개입 강도 축 하나를 강제하는 것이므로, 더 나은 축이 보이면 그쪽을 쓴다.

### 3. 시안 N개 생성 (기본 3)
각 시안마다:

- **이름** — 베팅이 드러나는 별칭 (예: "준비 체크리스트형", "기한 타임라인형")
- **가설** — "~다"로 끝나는 단언 한 문장. 이 시안이 옳다면 참인 믿음
- **우선 타겟** — 어느 사용자/상황에서 이기는가
- **분기 화면별 선택** — 분기 지점마다 이 시안이 택하는 방향 (레퍼런스 차용 출처 표기)
- **이길 수 있는 이유 / 리스크** — 리스크는 진짜여야 한다. 모든 시안의 리스크가 비슷하면 충분히 안 갈라진 것
- **검증할 것** — 프로토타입/테스트로 확인할 질문 1~2개

### 4. 컨펌 정지점
가설·축·분기 매트릭스를 텍스트 요약으로 보여주고 사용자 확인 후 Phase 2로 간다. 방향이 틀렸을 때 보드를 다시 그리는 비용이 크기 때문이다. 단, 사용자가 "바로 만들어줘"라고 했거나 레퍼런스 보드에 방향 분기가 이미 확정 수준으로 정리돼 있으면 멈추지 않고 진행한다.

## Phase 2 — 시안 보드 렌더링

단일 HTML 파일 하나, 외부 의존성 0 (웹폰트·CDN·JS 라이브러리 금지). 구조는 위에서 아래로:

1. **헤더** — 제목, 입력 문서 경로, 스코프(프레임 폭, 시안 수, 공통 처리 항목)
2. **가설 카드 N장** — 시안명 태그 + 가설 + 베팅 근거 + 리스크. 나란히 그리드
3. **분기 매트릭스** — 행=분기 지점, 열=시안. 한눈에 "무엇이 어떻게 갈렸나"
4. **화면 행** — 행=화면(분기 지점), 열=시안. **같은 화면을 시안끼리 나란히** 두는 것이 핵심 — 시안별 세로 나열이 아니라 화면별 가로 비교여야 차이가 보인다
5. **푸터** — 시안 간 공통 사항(분기 안 한 것과 그 이유), 다음 단계(시안 선택 또는 교배 → working-prototype), 제약 고지

프레임 규칙:
- 기본 375px 모바일 프레임 (입력 문서가 다른 플랫폼이면 그에 맞춤). 분기가 실재하는 화면만 그린다 — 시안당 4~6장이 적정. 전 플로우를 그리는 것은 프로토타입의 일이다
- 모든 프레임에 캡션: **시안번호-화면명 + 무엇이 다른지 한 문장 + 차용 레퍼런스 + 트레이드오프**. 캡션 없는 와이어는 보는 사람이 차이를 못 읽는다
- 공통 요소(진행 표시, 상단바 등)는 시안 간 픽셀까지 동일하게 — 차이가 분기 지점에만 있어야 비교가 성립한다
- 카피는 placeholder("텍스트 들어갈 자리")가 아니라 실제 후보 카피로 쓴다. 시안의 가설은 카피 톤에서도 드러나야 한다 (디자인시스템 Voice가 있으면 준수)

## 마무리

보드를 브라우저로 열어주고, 다음을 짧게 보고한다:
- 각 시안이 어느 상황에서 이기는지 한 문장씩
- 권장 프로토타입 우선순위와 이유 — 단정하지 말고 추천으로
- **교배 가능성**: 시안은 패키지가 아니다. "2안의 진입 + 1안의 복구"처럼 화면 단위 조합이 최종안이 되는 경우가 흔하므로, 그럴듯한 교배 조합 1개를 예시로 제안한다

다음 단계는 사용자의 시안 선택(또는 교배) → working-prototype 핸드오프다.
