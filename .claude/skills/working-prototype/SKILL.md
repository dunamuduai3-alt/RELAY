# Working Prototype (돌아가는 프로토타입)

기획을 **손으로 만져지는 프로토타입**으로 옮기는 스킬입니다. PRD·와이어프레임·디자인시스템을 입력받아, **의존성 0의 단일 HTML 파일 한 장**을 만듭니다. 정적 시안이 아니라 핵심 플로우가 실제로 클릭되고 화면이 전환되는, 브라우저로 바로 열리는 결과물입니다.

핵심 철학: **"돌아간다"가 전부다.** 좋은 프로토타입은 예쁜 한 장이 아니라, 데모에서 사람이 직접 눌러보며 "아 이렇게 흐르는구나"를 느끼게 하는 것입니다. 보여줄 단 하나의 플로우를 끝까지 동작하게 만들되, 나머지는 솔직하게 비워두세요.

## 산출물 계약 (파이프라인 입력)

이 스킬은 UX 파이프라인의 5번 단계입니다. 앞 단계 산출물을 입력으로 받습니다:

- **PRD** (2번 산출) — 무엇을·왜 만드는가, 핵심 플로우, 성공 기준. → 이게 프로토타입이 보여줘야 할 **하나의 플로우**를 정합니다.
- **와이어프레임 3종** (4번 산출, 있으면) — 화면 구조·배치의 출발점.
- **디자인시스템** (`assets/design-system.md`의 9섹션 형식) — 색·타이포·간격·컴포넌트·모션·보이스·안티패턴. 채워진 실물 예시는 `assets/design-system-example.md`(거래소 A, 라이트 테마) 참고.

**입력이 다 갖춰지지 않아도 시작하세요.** 없는 건 다음처럼 처리합니다:

- PRD가 없으면 → 사용자에게 "무슨 플로우를 보여줄지" 1-2개만 짧게 묻고 진행. 발굴 단계면 `product-brief` 스킬로 보내세요.
- 와이어가 없으면 → PRD와 디자인시스템만으로 합리적 레이아웃을 잡고 진행.
- 디자인시스템이 없으면 → `assets/design-system.md`를 골격으로 **최소 토큰(색·타이포·간격)만 빠르게 합의하거나** 추론해서 채우고, 추론한 사실을 명시. 기존 앱을 본떠야 하면 스크린샷을 받아 9섹션을 채우세요.

## 산출물 계약 (파이프라인 출력 — 6번 "스냅 & UX Lint"의 입력)

산출물은 두 파일입니다. HTML만 내고 끝내지 마세요:

1. **`[주제]-prototype.html`** — 의존성 0 단일 HTML. 핵심 플로우 동작 + 상태 4종 토글(3단계 참고).
2. **`[주제]-notes.md`** — 동반 문서 한 장, 두 섹션 필수:
   - **Design Decisions**: 화면의 주요 결정마다 "어떤 PRD 가설/와이어 선택 근거에서 나왔는가"를 1:1로 연결합니다. 추론·창작으로 메운 부분(디자인시스템에 없던 컴포넌트, 임의 레이아웃 판단)도 여기 명시하세요. 추적선 **PRD의 가설 → 화면 결정 → lint 검수 대상**이 파이프라인의 등뼈입니다.
   - **Lint Handoff**: 6번 단계가 스냅 떠야 할 **화면 × 상태 목록**(예: "주문 화면 — 성공/로딩/에러"), 알려진 한계·스텁 목록, 점검 제외 영역. 이게 없으면 6번 담당이 뭘 봐야 할지 다시 물어보게 됩니다.

## 작성 흐름

### 1단계: 보여줄 "하나의 플로우"를 먼저 못 박는다

데모의 성패는 범위에서 갈립니다. 앱 전체를 만들려 하지 마세요. PRD에서 **가장 중요한 단일 플로우 하나**(예: "로그인 → 재인증 → 거래")를 골라 그것만 끝까지 동작하게 만드세요. 화면 3~5개면 충분합니다. 나머지 진입점은 보이되 눌러도 "준비 중"으로 두는 게, 열 개를 반쯤 만드는 것보다 낫습니다.

골랐으면 사용자에게 "이 플로우를 중심으로 만들겠다"고 한 줄로 확인하고 진행하세요.

### 2단계: 디자인시스템을 토큰처럼 따른다 — 손으로 색을 짓지 않는다

`design-system.md`의 9섹션을 **그대로** 따르세요. 색·간격·라운드·폰트를 임의로 만들지 말고 정의된 값을 쓰세요. 특히:

- **CSS 변수로 토큰을 선언**하고(`--color-primary` 등) 전체가 그걸 참조하게 하세요. 그래야 일관되고, 나중에 토큰만 바꿔도 전체가 따라옵니다.
- **9번 Anti-patterns 섹션을 반드시 지키세요.** 거기 적힌 금지사항(예: "금액 반올림 금지")이 품질의 마지노선입니다.
- 디자인시스템에 없는 컴포넌트가 필요하면, 기존 토큰(색·간격·radius)으로 일관되게 만들고 그 사실을 notes의 Design Decisions에 적으세요.

### 3단계: 진짜로 동작하게 만든다 — 단일 self-contained HTML

산출물은 **외부 의존성 0의 `.html` 파일 한 장**입니다. 더블클릭하면 그냥 열려야 합니다.

- CSS는 `<style>`에 인라인, 상호작용은 바닐라 JS `<script>`로. **CDN·프레임워크·빌드 스텝 금지** — 오프라인에서, 발표장 와이파이 없이도 열려야 합니다.
- "돌아간다"의 의미: 버튼 클릭 → 화면/상태 전환, 입력 → 반응, 탭 전환이 **실제로** 됩니다. JS로 화면(섹션)을 show/hide 하는 SPA 패턴이면 충분합니다.
- **상태 4종을 토글 가능하게.** 핵심 플로우의 각 주요 화면은 **로딩 / 빈 / 에러 / 성공** 상태를 갖추고, 데모 중 강제로 전환할 수 있는 컨트롤을 넣으세요(예: 우상단 작은 dev 스위치, 또는 키보드 1·2·3·4). 자연 플로우로 도달 가능한 상태라도 토글은 넣습니다 — 6번 UX Lint가 상태별 스냅을 떠야 하고, 발표에서 "에러도 설계했다"를 3초 만에 보여주는 장치이기도 합니다. 해당 화면에 의미 없는 상태(예: 빈 상태가 성립 안 하는 화면)는 만들지 말고 notes에 "해당 없음"으로 명시하세요.
- 데이터는 하드코딩된 그럴듯한 목업으로. 단, **무엇이 가짜인지 코드 주석이나 화면에 표시**하세요(예: 더미 잔고). 실제로 안 되는 버튼은 비활성 처리하거나 "준비 중" 토스트로 — 눌렀는데 아무 일도 없으면 데모에서 깨집니다.
- 모바일 앱 플로우면 화면을 폰 프레임(고정 폭, 예 375px) 안에 넣어 맥락을 주세요.

### 4단계: 능동적으로 빈틈을 짚는다

받아쓰기에 그치지 마세요. 만들면서 발견한 것을 먼저 말하세요: "PRD엔 A 화면이 있지만 이 플로우엔 불필요해 뺐다", "이 단계는 실제론 서버 응답을 기다려야 해서 로딩 상태를 넣었다", "와이어와 달리 버튼을 여기 두는 게 엄지 도달에 낫다". UX 관점의 판단을 더하는 게 이 스킬의 값어치입니다. 이렇게 짚은 판단들이 그대로 notes의 Design Decisions 재료가 됩니다.

### 5단계: 저장하고 다음 단계를 제안한다

`[주제]-prototype.html`과 `[주제]-notes.md` 두 파일을 저장하세요(예: `login-reauth-prototype.html` + `login-reauth-notes.md`). notes는 출력 계약의 두 섹션(Design Decisions / Lint Handoff)을 갖춰야 합니다. 저장 후:

- **경로를 알리고**, 어떻게 열어보는지(더블클릭 또는 브라우저로 열기) 한 줄.
- **무엇이 동작하고 무엇이 목업/스텁인지** 짧게 정리 — 데모 전에 알아야 발표가 안 깨집니다. (긴 버전은 notes의 Lint Handoff에)
- **자연스러운 다음 단계** 제안: 파이프라인 6번 "스냅 & UX Lint"(Nielsen 10 휴리스틱)로 이 프로토타입을 점검할지 — notes의 Lint Handoff 섹션을 그대로 넘기면 됩니다.

## 톤·품질

- 프로토타입은 코드 미인대회가 아니라 **생각을 만져보게 하는 도구**입니다. 깔끔하되 과하게 엔지니어링하지 마세요 — 한 파일이면 충분합니다.
- **솔직함이 핵심입니다.** 가짜 데이터, 스텁된 버튼, 생략한 화면을 숨기지 말고 명시하세요. "다 됩니다"라고 말한 뒤 데모에서 깨지는 것보다, "이 플로우는 되고 저긴 스텁"이라고 미리 말하는 게 백 배 낫습니다.
- 한국어로 간결하게. 화면 안 카피는 디자인시스템의 7번 Voice를 따르세요.

## 주의: 공개·익명화 제약

프로토타입을 외부 공개·발표에 쓸 경우, 입력에 **익명화 제약이 걸린 자사 제품/데이터**가 있는지 먼저 확인하세요(예: 특정 거래소를 '거래소 A'로 표기해야 하는 경우). 제약이 있으면 화면 카피·브랜드 요소·더미 데이터 전반에 일관되게 적용하세요.
