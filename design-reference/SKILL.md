# Design Reference — 코멘트 붙은 레퍼런스 보드

디자인하기 전에, 또는 하다가 막혔을 때 "다른 서비스는 이걸 어떻게 풀었지?"에 답하는 스킬.
핵심은 **수집이 아니라 판단 재료**다. 스크린샷 20장을 쌓아주는 건 도움이 안 된다 —
레퍼런스마다 "왜 참고할 만한지 / 우리 케이스에 어떻게 적용할지" 코멘트가 붙어 있어야
디자이너가 보자마자 결정에 쓸 수 있다. 코멘트 없는 레퍼런스는 산출물에 넣지 않는다.

## 두 가지 모드

- **범용 모드 (기본)** — 사용자가 지목한 화면/플로우/패턴을 주제로 리서치한다.
- **파이프라인 모드** — 입력에 PRD 파일이 있으면(또는 사용자가 "PRD 기반으로"라고 하면) 발동.
  PRD의 가설·화면 시나리오를 추출해 각각에 벤치마크 패턴을 매핑하고,
  다음 단계(와이어프레임 3종)에 바로 먹일 핸드오프 섹션을 추가한다. 아래 [파이프라인 모드](#파이프라인-모드) 참고.

## 0. 스코프 잡기

리서치 품질은 검색어가 아니라 스코프에서 갈린다. 시작 전에 다음이 명확한지 확인하고,
**불명확한 것만** 짧게 묻는다 (이미 대화나 입력 파일에 답이 있으면 묻지 않는다):

1. **어떤 화면/플로우인가** — "온보딩"처럼 넓으면 "온보딩 중 어느 순간? (가입 직후 가치 전달 / 권한 요청 / 정보 입력)"으로 좁힌다.
2. **플랫폼** — mobile / desktop. 모르면 묻기보다 작업 맥락(프로토타입이 375px이면 mobile)으로 추론한다.
3. **꼭 보고 싶은 서비스가 있는가** — 있으면 그 서비스를 우선 검색하되, 거기에만 갇히지 않는다.

스코프가 잡히면 리서치할 **패턴 후보 2~5개**를 먼저 선언하고 시작한다.
(예: "막힘 화면" 주제라면 → ①막힌 사유 설명 ②남은 단계/예상 시간 표시 ③복구 경로 제시)
이 선언이 검색 계획이자 최종 산출물의 목차가 된다.

## 1. 레퍼런스 수집

### Lazyweb — 실물 스크린샷 (1차 소스)

`lazyweb_search` MCP 도구로 실제 서비스 스크린샷을 받는다. 실물 증거가 붙은 레퍼런스가
아티클 속 묘사보다 훨씬 설득력 있으므로 이게 기본 소스다.

- **패턴/화면 하나당 검색 하나.** "fintech onboarding error states progress" 같은 욕심 검색 하나보다
  "identity verification progress indicator", "KYC error recovery" 같은 좁은 검색 여러 번이 낫다.
- `platform`("mobile"/"desktop")을 지정하고, 특정 서비스를 봐야 하면 `company:"앱이름"`을 쿼리에 넣는다.
- 같은 쿼리 반복 금지(결과가 결정적). 더 보려면 `offset`으로 페이징하고, 응답의 `warnings`/`coverage`를 읽는다.
- 결과 이미지는 `curl`로 산출물 폴더의 `images/`에 내려받는다. 파일명은 `<서비스>-<패턴>-<n>.png` 형태로.
- 검색 결과가 빈약한 패턴은 솔직하게 "실물 레퍼런스를 못 찾았다"고 적는다. 억지로 무관한 스크린샷을 끼워 넣지 않는다.

### 웹 검색 — 근거 아티클 (2차 소스)

스크린샷은 "무엇을 했는지"는 보여주지만 "왜 그게 좋은지"는 말해주지 않는다.
코멘트의 근거를 보강할 때 웹 검색을 쓴다: NN/g, Baymard, 디자인 시스템 문서, 사례 분석 글 등.

- 아티클은 코멘트에 인용할 때만 가치가 있다. 읽고 버릴 거면 검색하지 않는다.
- 인용하면 반드시 출처 URL을 남긴다. 국내 서비스(토스·카카오뱅크 등) 사례는 네이버 검색이 더 잘 나올 수 있다.

## 2. 코멘트 규율 — 이 스킬의 핵심

레퍼런스마다 세 줄을 쓴다. 사실/해석/제안을 섞지 않는 게 요점이다 —
섞이면 읽는 사람이 "이게 화면에 진짜 있는 건지, 네 추측인지" 구분할 수 없게 된다.

1. **보이는 것 (사실)** — 스크린샷에 실제로 있는 요소만. "Revolut은 단계 4개를 상단에 도트로 표시하고 현재 단계에 라벨을 단다."
2. **왜 참고할 만한가 (해석)** — 어떤 문제를 어떻게 푸는 패턴인지. 아티클 근거가 있으면 여기서 인용.
3. **우리 케이스 적용 포인트 (제안)** — 지금 작업에 어떻게 가져올지. 그대로 베끼라는 게 아니라 "이 원리를 이 화면에" 수준으로.

지킬 것:
- 근거 없이 "베스트 프랙티스다", "사용자들이 선호한다"라고 단정하지 않는다. 근거가 있으면 출처를 달고, 없으면 "~로 보인다"로 쓴다.
- 좋은 점만 쓰지 않는다. 레퍼런스의 **한계나 우리 맥락과 안 맞는 부분**도 한 줄 적어주면 판단 재료로서 가치가 올라간다.

## 3. 패턴으로 묶기

산출물은 **서비스별이 아니라 패턴별**로 구성한다. "토스 섹션, 레볼루트 섹션"이 아니라
"진행 표시 패턴 (토스, 레볼루트, N26)" 식이다. 디자이너가 필요한 건 서비스 도감이 아니라
"이 문제를 푸는 방법이 몇 가지 있고 각각 트레이드오프가 뭔지"이기 때문이다.
패턴 섹션 끝에는 한 줄 비교("A안은 공간을 아끼지만 단계명이 안 보임, B안은 그 반대")를 단다.

## 4. 출력 계약

`<주제>-references/` 폴더 하나에 전부 담는다:

```
<주제>-references/
├── references.md   ← 작업 입력용 (다음 단계가 읽는 파일)
├── board.html      ← 공유/발표용 보드 (더블클릭으로 열림)
└── images/         ← 다운로드한 스크린샷
```

### references.md 구조

```markdown
# [주제] 레퍼런스
## 요약 — 스코프, 본 패턴 목록, 핵심 발견 3줄
## 패턴 1: [패턴명]
### [서비스명] — ![](images/...)
- 보이는 것: …
- 왜 참고: … (출처: [링크])
- 적용 포인트: …
(패턴 끝에 한 줄 비교)
## 패턴 2: …
## 적용 제안 — 우리 케이스에 추천하는 방향 + 이유
## 출처 목록
```

### board.html 스펙

- **의존성 0 단일 HTML** — 외부 CSS/JS/폰트 없음, 인라인 스타일, 더블클릭으로 열림.
- 이미지는 `images/` 상대 경로 참조 (폴더째 옮기면 같이 따라감).
- 라이트 테마, 패턴별 섹션 헤더 + 카드 그리드. 카드 = 스크린샷 + 서비스명 + 코멘트 3줄 + 출처 링크.
- 보드는 references.md의 **뷰**다 — 내용을 따로 쓰지 말고 같은 내용을 보기 좋게 배치만 한다. 둘이 어긋나면 md가 정본.

## 파이프라인 모드

PRD가 입력이면 위 작업에 다음을 더한다:

- **스코프 잡기를 PRD에서 시작한다.** 가설(solution hypothesis)과 화면 시나리오를 추출해 패턴 후보를 도출하고, 사용자에게 "이 가설들에서 이 패턴들을 리서치하겠다"고 확인받는다.
- **추적선을 유지한다.** 각 패턴 섹션 첫머리에 어느 PRD 가설에 대응하는지 명시한다 (`> PRD 가설 H1: …에 대응`). 파이프라인 전체의 추적선(PRD 가설 → 벤치마크 패턴 → 와이어 결정 → lint 검수)에서 이 스킬이 두 번째 고리다.
- **references.md 끝에 `## 와이어프레임 핸드오프` 섹션을 추가한다.** 다음 단계(와이어 3종)가 그대로 쓸 수 있게: 가설별로 "시도해볼 만한 패턴 방향 2~3개 + 각 방향의 트레이드오프 한 줄". 와이어 3종이 서로 다른 방향이 되도록 방향들이 실제로 갈라져 있어야 한다.

## 익명화 가드

자사 서비스 관련 작업이면: 공유·발표 가능성이 있는 산출물(references.md, board.html, 파일명, 폴더명)에서
자사 서비스 실명을 쓰지 않고 **'거래소 A'**로 표기한다. 경쟁/타사 서비스 레퍼런스는 실명 그대로 쓴다.
헷갈리면 작업 시작 시 사용자에게 이 산출물이 외부 공유용인지 한 번 확인한다.
