# RELAY — UX 파이프라인 노하우

거래소 A · KYC(고객확인) 재이행 해커톤에서 **6단계 디자인 플로우**를 실제로 돌리며 쌓은, 단계별 노하우와 **바로 복붙 가능한 프롬프트** 모음입니다. (정답은 없습니다 — 경험 공유가 목적이에요.)

> 원본 시트: [노하우-UX RELAY (Google Sheets)](https://docs.google.com/spreadsheets/d/1pI3joKJomT3NoDD2PWYaP2wNGpVXosycas8znVMM-2Q/edit)

**범례** — `①~⑥` = 핵심 디자인 플로우 6단계(문제발굴 → PRD → 벤치마크·레퍼런스 → 시안분기 → 프로토타입 → UX Lint) · `◆` = 플로우를 가로지르는 지원 활동(디자인시스템·TO-BE 조립·협업·스킬화·인프라)

---

## 한눈에 보는 핵심 노하우

| 구분 | 핵심 노하우 |
|---|---|
| 컨셉 | 기성 툴을 쓰는 데 그치지 않고, 디자인 워크플로우를 **재사용 스킬 파이프라인**으로 만든다. |
| 출발 | 머릿속 가설이 아니라 **실데이터(자사 리뷰 560여 건)**에서 문제를 발굴한다. |
| 근거 | 화면 그리기 전, 레퍼런스를 모아 **PRD 가설 ↔ 참고 패턴을 1:1로 매핑**해 둔다. |
| 분기 | 화면을 확정하기 전에 **UX 가설 N종으로 갈라** 트레이드오프를 눈으로 비교한다. |
| 자동 | 디자인시스템·화면을 **Figma MCP로 토큰 바인딩**해 코드와 디자인을 한 소스로 묶는다. |
| 검증 | 만든 사람이 곧장 휴리스틱 심사역으로 전환해(직접 만든 `snap-ux-lint`) **날 선 각도로 점검**한다. *(우리의 핵심 단계)* |
| 도구화 | 핵심 단계를 **재사용 스킬 3종**으로 박제한다 — `review-mining`·`design-variants`·`snap-ux-lint` (일부 Codex 호환). |
| 지속 | 안 바뀌는 제약을 **AI 지속 메모리**에 박아 세션이 끊겨도 일관성을 유지한다. |

## 단계별 노하우

### 1 · ① 문제발굴
- **상황** — 무엇을 만들지 아직 모르고, 실제 사용자 불만에서 문제를 찾고 싶을 때
- **쓴 도구** — `review-mining` 스킬 (직접 제작) → `product-brief` 스킬
- **노하우** — 임의 가정으로 기획을 시작하지 않았습니다. 직접 만든 review-mining 스킬로 구글플레이·네이버 카페·디시 등 자사 앱 리뷰 560여 건을 CSV째 넣어, 개인정보 마스킹 후 최근 12개월·UX 신호 위주로 반복 페인포인트를 빈도×심각도로 군집화했더니 '본인확인(KYC) 재이행 단계에서 막혀 이탈'이라는 진짜 문제가 주제를 정해줬습니다. 머릿속 가설이 아니라 증거가 주제를 고르게 한 게 핵심. 스킬 끝단의 PRD 핸드오프(문제 후보·검증 질문·해결 가설)가 다음 단계로 바로 이어졌습니다.
- **산출물** — 리뷰 마이닝 리포트(.html) + 원본 raw CSV(마스킹) + PRD 핸드오프 + 발표용 브리프 덱(.html)
- **복붙 프롬프트**
  ```
  첨부한 자사 앱 스토어·카페 리뷰에서 사용자 페인포인트를 발굴해줘. 임의로 지어내지 말고
  실제 문장을 근거로 최근 12개월·UX 신호 위주로 군집화하고, 빈도×심각도와 근거 인용을 붙여
  핵심 문제 1개로 좁혀줘. 개인정보는 마스킹.
  ```

### 2 · ② PRD
- **상황** — 문제 가설이 단단해져서 요구사항(PRD)으로 확장할 때
- **쓴 도구** — `product-brief` 스킬 + 웹 리서치
- **노하우** — 같은 흐름에서 브리프를 PRD로 확장하되, KYC·재인증은 규제 영역이라 '왜 이 단계가 법적으로 존재하는가'를 먼저 리서치해 PRD에 건드리면 안 되는 제약으로 박아뒀습니다. 이렇게 해두니 '마찰을 줄이자'가 규제 위반으로 새지 않고, 제약 안에서 줄일 수 있는 마찰만 요구사항이 됐습니다.
- **산출물** — 정책 리서치(.md) + 재이행 PRD(.md) (제약 / 핵심 플로우 / 성공 기준 분리)
- **복붙 프롬프트**
  ```
  이 브리프를 PRD로 확장해줘. 먼저 KYC 재이행이 법적으로 왜 필요한지 리서치해서
  '바꾸면 안 되는 제약'으로 명시하고, 그 제약 안에서 줄일 수 있는 마찰만 요구사항으로 잡아줘.
  ```

### 3 · ③ 벤치마크
- **상황** — 자사 현행 화면과 경쟁사는 이 플로우를 어떻게 풀었나 볼 때
- **쓴 도구** — Claude Code (이미지 일괄 읽기)
- **노하우** — 경쟁 거래소 재인증 플로우 캡쳐 24장과 자사 현행(AS-IS) 캡쳐 11장을 통째로 넣어, 다시 그리지 않고 '단계·마찰 지점'만 명세로 추출했습니다. 재사용할 자산(디자인시스템 등)은 직접 만들되, 외부·일회성 화면은 읽고 명세만 뽑는 게 훨씬 빨랐습니다.
- **산출물** — 경쟁사 플로우 분석(.md) + AS-IS 단계·마찰 정리표
- **복붙 프롬프트**
  ```
  첨부한 경쟁 거래소 재인증 플로우 캡쳐를 화면 단위로 읽고, 단계별 흐름과 사용자 마찰 지점만
  표로 정리해줘. 픽셀을 베끼지 말고 구조와 의사결정만 추출해.
  ```

### 4 · ③ 레퍼런스
- **상황** — 화면을 그리기 전 참고 패턴을 모아 근거를 만들 때
- **쓴 도구** — `design-reference` 스킬 (Lazyweb + 웹)
- **노하우** — 빈손으로 와이어를 그리는 대신, 다른 앱들이 본인인증·진행표시·막힘(에러) 화면을 어떻게 풀었는지 실제 스크린샷으로 모아 '코멘트 붙은 레퍼런스 보드'로 만들었습니다. PRD 가설마다 참고 패턴을 1:1로 매핑해두니, 와이어를 그릴 때 '왜 이렇게?'에 항상 근거가 있었습니다.
- **산출물** — 코멘트 보드(.html, 단일파일 공유용) + 플로우 보드 + references(.md)
- **복붙 프롬프트**
  ```
  KYC 재인증 화면을 그리기 전에, 다른 앱들은 본인인증·진행표시·막힘 화면을 어떻게 했는지
  레퍼런스 보드로 만들어줘. PRD 가설마다 참고할 패턴을 매핑하고 한 줄 코멘트를 달아.
  ```

### 5 · ④ 시안분기
- **상황** — 한 화면을 어떻게 그릴지 방향이 여러 갈래일 때
- **쓴 도구** — `design-variants` 스킬 (직접 제작)
- **노하우** — '어떻게 그릴까'를 바로 정하지 않고, 분기 축(예: 한 번에 다 보여주기 ↔ 단계 쪼개기)을 먼저 정한 뒤 서로 다른 UX 가설 3종으로 갈라 한 장의 비교 보드(375px 프레임 나란히)로 만들었습니다. 고르기 전에 트레이드오프가 눈에 보이는 게 이 단계의 값어치.
- **산출물** — 가설 카드 + 분기 매트릭스 + 시안 3종 비교 보드(.html) (+ 진행표시 등 부분요소 비교)
- **복붙 프롬프트**
  ```
  이 화면을 서로 다른 UX 가설 3종으로 갈라줘. 분기 축을 먼저 정하고, 각 안의 가설·장단점
  카드와 함께 375px 프레임으로 나란히 비교되는 단일 HTML 보드로 만들어.
  ```

### 6 · ◆ 디자인시스템
- **상황** — 디자인시스템을 코드 토큰에서 Figma로 옮길 때
- **쓴 도구** — Figma MCP (claude.ai Figma)
- **노하우** — 색·타이포를 손으로 짓지 않았습니다. 검증된 디자인시스템 프리셋(shadcn 토큰)을 차용해 라이트+다크 토큰 명세(.md)로 먼저 고정한 뒤, Figma MCP로 변수·컴포넌트를 Figma 파일에 자동 생성했습니다. 코드와 디자인이 같은 토큰을 공유해, 한쪽만 바꿔도 어긋나지 않습니다.
- **산출물** — 디자인시스템 명세(.md) + 변수·컴포넌트가 들어간 Figma 파일
- **복붙 프롬프트**
  ```
  이 토큰 명세(라이트+다크)를 Figma 변수와 컴포넌트로 만들어줘. 색은 임의로 만들지 말고
  명세 값만 바인딩하고, 다크 모드는 같은 변수의 모드로 연결해.
  ```

### 7 · ◆ TO-BE 조립
- **상황** — TO-BE 화면 전체를 Figma로 조립할 때
- **쓴 도구** — Figma MCP
- **노하우** — 확정 시안 + 디자인시스템 토큰으로 재이행 플로우 TO-BE 화면을 Figma에 화면 단위로 조립했습니다. 핵심은 하드코딩 값 대신 변수 바인딩 — 토큰만 바꿔도 전 화면 색·간격이 따라옵니다.
- **산출물** — 토큰 바인딩된 TO-BE 와이어프레임 Figma 파일 (재이행 플로우 전 화면)
- **복붙 프롬프트**
  ```
  확정 시안과 디자인시스템 변수를 써서 재이행 플로우 TO-BE 화면을 Figma에 화면 단위로
  조립해줘. 색·간격은 전부 변수 바인딩으로, 하드코딩 금지.
  ```

### 8 · ◆ 협업
- **상황** — 팀 디자이너 와이어에 다시 안 그리고 피드백만 줄 때
- **쓴 도구** — Figma MCP
- **노하우** — 동료가 그린 와이어 위에 내가 새로 그리는 대신, UX 원칙별 포스트잇(코멘트)을 붙였습니다. 결정권은 그린 사람에게 두되 근거(휴리스틱)는 화면에 남기는 방식 — 협업에서 마찰이 가장 적었습니다.
- **산출물** — 와이어 파일 위에 UX 원칙 코멘트 포스트잇 N장
- **복붙 프롬프트**
  ```
  이 Figma 와이어 3종에 Nielsen 휴리스틱 기준으로 개선 포인트를 포스트잇으로 붙여줘.
  다시 그리지 말고 코멘트만, 어떤 원칙인지 이름과 함께.
  ```

### 9 · ⑤ 프로토타입
- **상황** — 정적 시안을 넘어 실제로 클릭되는 데모가 필요할 때
- **쓴 도구** — claude.ai/design
- **노하우** — 확정 와이어(Figma)·디자인시스템을 넘겨 claude.ai/design에서 핵심 플로우 하나가 실제로 클릭되는 프로토타입을 바로 생성했습니다. 빌드·배포 없이 프롬프트로 화면을 만들고 공유 링크로 즉시 돌려볼 수 있어, 정적 시안에서 '만져지는 데모'로 가는 시간이 가장 짧았습니다. 화면별 상태(로딩/빈/에러/성공)도 프롬프트로 지시해 '에러까지 설계했다'를 보여줬고, 범위를 한 플로우로 좁힌 게 데모가 안 깨진 비결.
- **산출물** — 돌아가는 프로토타입(.html) (의존성 0, 클릭되는 플로우 + 상태 토글)
- **복붙 프롬프트**
  ```
  확정 와이어(Figma)와 디자인시스템으로 핵심 플로우 하나가 실제로 클릭되는 인터랙티브
  프로토타입을 만들어줘. 화면별 상태(로딩/빈/에러/성공)까지 분기하고, 범위는 한 플로우로 좁혀.
  ```

### 10 · ⑥ UX Lint *(핵심 단계)*
- **상황** — 내가 만든 걸 냉정하게 검수해야 할 때
- **쓴 도구** — `snap-ux-lint` 스킬 (직접 제작)
- **노하우** — 만든 사람이 곧장 '심사역'으로 역할을 바꿔, 직접 만든 snap-ux-lint 스킬로 점검했습니다. 사용자 목표·제약(PRD/정책)을 먼저 받고 → 플로우 스냅샷 → 플로우/화면별 Nielsen 10 휴리스틱 린트 → 심각도·우선순위·확신도 → 마지막에 스크린샷 위에 콜아웃을 단 '주석 UX 스냅샷'까지 냅니다. '좋게 포장하지 마'를 명시해 자기 편향을 줄인 게 핵심 — 우리 파이프라인이 진짜 강해지는 지점은 여기였습니다.
- **산출물** — 플로우 스냅샷 + 화면별 휴리스틱 위반 목록(심각도·우선순위) + 수정 리스트 + 주석 UX 스냅샷
- **복붙 프롬프트**
  ```
  이 플로우의 사용자 목표와 제약(PRD/정책)을 먼저 묻고, 그다음 화면×상태를 Nielsen 10
  휴리스틱으로 린트해줘. 위반마다 근거·영향·심각도·우선순위를 붙이고, 좋게 포장하지 말고
  가장 약한 곳부터. 마지막에 스크린샷에 주석 단 UX 스냅샷으로 정리해.
  ```

### 11 · ◆ 스킬화
- **상황** — 같은 작업을 반복하다 워크플로우 자체를 도구로 박제할 때
- **쓴 도구** — skill-creator
- **노하우** — 매번 같은 지시를 반복하는 대신, 파이프라인의 핵심 단계를 각각 재사용 스킬로 박제했습니다 — 리뷰 마이닝(review-mining)·시안 N종 분기(design-variants)·UX 린트(snap-ux-lint). 다음 화면·다음 프로젝트에서 한 단어로 호출되고, 일부는 Codex에서도 호출되도록(agents 설정) 만들었습니다. 기성 툴을 쓰는 사람에서 자기 도구를 만드는 디자이너로 넘어간 순간.
- **산출물** — 커스텀 스킬 3종(review-mining·design-variants·snap-ux-lint)
- **복붙 프롬프트**
  ```
  방금 우리가 반복한 이 작업을 재사용 가능한 스킬로 만들어줘.
  트리거 문장 / 입력·출력 계약 / 단계별 작성 흐름을 SKILL.md로.
  ```

### 12 · ◆ 인프라
- **상황** — 세션이 바뀌어도 프로젝트 제약·결정을 잊지 않게 하고 싶을 때
- **쓴 도구** — Claude Code 지속 메모리(memory)
